# 动态规划功能说明

## 概述

本次更新为 QuantManus 添加了**动态规划（反思机制）**功能，解决了原系统执行计划过于僵化的问题。

## 问题背景

### 原有问题
在原系统中，规划模式存在以下问题：

1. **规划阶段**：LLM 一次性生成完整计划
2. **执行阶段**：严格按照预设步骤执行，不会根据中间结果动态调整

**典型场景**：
```
步骤1: 读取 requirements.md 文件
步骤2: 分析数据
步骤3: 生成报告
```

如果 `requirements.md` 中包含 "请先读取 config.json 文件"，系统会忽略这个指示，直接执行步骤2。

## 解决方案

### 实现的功能：反思重规划机制

在每个步骤执行完成后，系统会：

1. **分析执行结果**：检查结果中是否包含新的指令或依赖
2. **智能判断**：判断是否需要调整后续计划
3. **动态插入**：如有必要，在当前步骤后插入新步骤
4. **继续执行**：按照调整后的计划继续执行

### 核心改进

#### 1. 在 `planner.py` 中添加反思方法

```python
def reflect_and_adjust_plan(self, plan: Plan, executed_step: Step) -> List[Step]:
    """
    反思步骤执行结果，并决定是否需要动态调整计划
    """
```

**功能**：
- 分析刚执行完的步骤结果
- 识别结果中的新指令（如 "请读取xxx文件"）
- 生成需要插入的新步骤
- 自动设置依赖关系

**判断规则**：
- ✅ 执行结果中明确提到 "请读取xxx文件"、"需要先处理xxx" → 插入新步骤
- ✅ 执行结果揭示新的依赖关系或前置条件 → 调整计划
- ❌ 执行结果只是正常数据或确认信息 → 无需调整

#### 2. 在 `plan_executor.py` 中集成反思机制

```python
# 在步骤执行成功后
if self.enable_dynamic_planning and hasattr(self.agent, 'planner'):
    new_steps = self.agent.planner.reflect_and_adjust_plan(plan, step)
    if new_steps:
        for new_step in new_steps:
            plan.add_step(new_step)
```

**特点**：
- 可通过 `enable_dynamic_planning` 参数控制是否启用
- 默认启用动态规划
- 新步骤自动依赖当前步骤

## 使用示例

### 场景：多层文件引用

**文件结构**：
```
requirements.md
  ↓ 提到
data_config.txt
  ↓ 提到
preprocessing_rules.json
```

**执行流程**：

1. **初始计划**（由LLM生成）：
   ```
   步骤1: 读取 requirements.md
   步骤2: 按照要求执行任务
   步骤3: 生成报告
   ```

2. **动态调整过程**：

   执行步骤1后，系统发现 requirements.md 中提到 data_config.txt
   ```
   🔄 反思机制触发
   📌 插入新步骤: 步骤4 - 读取 data_config.txt
   ```

   执行步骤4后，系统发现 data_config.txt 中提到 preprocessing_rules.json
   ```
   🔄 反思机制触发
   📌 插入新步骤: 步骤5 - 读取 preprocessing_rules.json
   ```

3. **最终执行顺序**：
   ```
   步骤1: 读取 requirements.md ✅
   步骤4: 读取 data_config.txt ✅ (动态添加)
   步骤5: 读取 preprocessing_rules.json ✅ (动态添加)
   步骤2: 按照要求执行任务 ✅
   步骤3: 生成报告 ✅
   ```

## 配置选项

### 启用/禁用动态规划

在创建 `SimpleAgent` 时，`enable_planning=True` 会自动启用动态规划。

如果需要禁用反思机制（保持原有行为），可以修改 `PlanExecutor` 初始化：

```python
# 在 agent.py 中修改
self.plan_executor = PlanExecutor(self, self.logger, enable_dynamic_planning=False)
```

### 调整反思灵敏度

在 `planner.py` 的 `_build_reflection_prompt` 方法中，可以调整判断规则：

```python
判断规则：
- 如果执行结果中明确提到"请读取xxx文件"、"需要先处理xxx"等指令，应该插入新步骤
- 如果执行结果揭示了新的依赖关系或前置条件，应该调整计划
- 如果执行结果只是正常的数据或确认信息，无需调整
```

## 性能影响

### 额外开销

- 每个步骤执行后多一次 LLM 调用（用于反思）
- 通常耗时：0.5-2秒/步骤
- 对整体执行时间影响：约增加 10-20%

### 优势

- 大幅提高任务执行的灵活性
- 减少因遗漏指令导致的失败
- 更符合人类的思维方式

## 技术细节

### 反思提示词结构

```
1. 原始任务描述
2. 刚执行完的步骤信息（包括结果）
3. 后续待执行的步骤列表
4. 可用工具列表
5. 判断规则和输出格式
```

### 新步骤生成规则

- **自动依赖**：新步骤默认依赖触发反思的步骤
- **ID分配**：从当前最大ID+1开始
- **插入位置**：添加到计划末尾，由依赖关系控制执行顺序

### 错误处理

- 反思过程出错 → 记录警告，继续原计划
- 解析失败 → 返回空步骤列表，不影响执行
- LLM调用失败 → 降级到普通执行模式

## 未来优化方向

1. **智能缓存**：相似场景复用反思结果
2. **优先级调整**：动态调整步骤执行顺序
3. **并行执行**：识别可并行的步骤
4. **学习机制**：记录调整历史，优化未来规划

## 常见问题

### Q: 反思机制会增加多少执行时间？
A: 每个步骤约增加 0.5-2秒，整体约 10-20%。对于复杂任务，提高的准确性远超时间成本。

### Q: 如何查看反思过程的日志？
A: 在日志中查找 "正在反思步骤" 和 "根据步骤X的结果，动态添加" 等关键词。

### Q: 反思机制会不会陷入无限循环？
A: 不会。新步骤只在执行后触发一次反思，且有最大步骤数限制。

### Q: 可以手动干预调整过程吗？
A: 当前版本自动执行。未来可以添加交互式确认机制。
